DEF-189 Keyboard input captured by calculator while typing in login fields Fixed P2 V1 ▸

Reporter: Cristian R. (Romania QA) · Filed: Feb 20, 2026 · Fixed by: frontend-team · Deployed: Feb 27, 2026 (v2 only)

Description

When entering a password in the login form, numeric key presses are intercepted by the calculator's global keyboard handler. For example, typing the password "password123" causes '123' to appear on the calculator display. Other keyboard shortcuts (Backspace, Escape, operators) are also captured, interfering with normal text editing in the password field.

Steps to Reproduce

1. Open TestCalc (do not click on the calculator). 2. Click into the Password field. 3. Type "password123". 4. Observe the calculator display — it now shows "123". 5. Press Backspace to correct a typo in the password — the calculator also deletes a digit.

Expected Result

Keyboard input in form fields should not affect the calculator.

Actual Result

The global document.addEventListener('keydown', ...) handler does not check whether the event target is a form input. All mapped keys (digits, operators, Backspace, Escape, Enter) are captured by the calculator regardless of focus context.

dev-sarah — Feb 22, 2026
Good catch. The keyboard handler was added without considering focus context. Added an active element check for the password field in v2. V1 will not be patched since it's in maintenance mode.
Resolution: Added focus detection to suppress calculator keyboard shortcuts when the password input field is active. Fix deployed to v2 only.

Note: V1 remains affected. No backport planned.
